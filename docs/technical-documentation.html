Resumo do sistema

Painel desktop moderno para snapshots e subvolumes Btrfs.

O Btrfs Manager é dividido em uma aplicação GTK/libadwaita sem privilégio, um helper privilegiado estreito e uma crate Rust compartilhada de domínio. O objetivo arquitetural é segurança operacional: descobrir corretamente, navegar em modo somente leitura, não modificar snapshots externos e manter rollback reversível.

01 · Visão Geral da Arquitetura

Estrutura, responsabilidades e fronteiras

A arquitetura gira em torno da fronteira de privilégio. Em builds instalados, a GUI fala com o helper pelo system bus (`org.btrfsmanager.Helper`) e o helper autoriza cada `HelperRequest` via Polkit antes de executar operações Btrfs.

crates/core

Domínio puro

Modelos, parsing de outputs, path safety, retenção, comparação e plano de rollback. Não conhece GTK nem Polkit.

crates/helper

Fronteira privilegiada

Executa operações Btrfs e mount/umount com validação de paths, API estreita baseada em `HelperRequest`, serviço D-Bus e ações Polkit por classe operacional.

crates/app

Experiência desktop

GUI GTK/libadwaita, busca, inventário, browse de snapshots, toasts de erro e chamada ao helper.

02 · Fluxo de Dados e Ciclo de Vida

Do evento da interface ao resultado operacional

Aplicação inicia

A GUI cria `UiState`, `ToastOverlay`, seletor de filesystem e lista principal. Se existem mounts gerenciados antigos, tenta cleanup seguro.

Discovery autorizado

`DiscoverFilesystems` passa pelo serviço D-Bus em builds instalados; a ação Polkit de discovery permite sessão ativa sem senha. Em desenvolvimento, fallback local só existe com `BTRFS_MANAGER_DEV_LOCAL_HELPER=1`.

Inventário

`ListSubvolumes` executa `btrfs subvolume list -u`, parseia linhas, classifica cada subvolume e retorna `SubvolumeInventory` estruturado.

Renderização

A GUI separa `Snapshot`/`ExternalSnapshot` de `Normal`/`SnapshotContainer`, mostrando `@snapshots` como container e snapshots reais na seção de snapshots.

Browse read-only

Ao abrir snapshot, resolve path direto ou monta top-level `subvolid=5`, cria bind mount read-only e abre o path com `xdg-open`.

Feedback e cleanup

Toasts mostram sucesso/erro; a row muda para estado montado; o usuário pode desmontar explicitamente.

Evento da GUI
  -> HelperRequest
  -> D-Bus org.btrfsmanager.Helper quando instalado
  -> autorização Polkit por classe de ação
  -> fallback local somente com BTRFS_MANAGER_DEV_LOCAL_HELPER=1
  -> parser/modelos do core classificam os dados
  -> SubvolumeInventory JSON
  -> linhas e ações na lista GTK
03 · Cérebro e Lógica de Decisão

Regras que protegem o sistema

Classificação

SubvolumeKind

`Normal`, `SnapshotContainer`, `Snapshot` e `ExternalSnapshot`. Essa classificação substitui heurísticas duplicadas na GUI.

→
Renderização

Seções corretas

Containers como `@snapshots` ficam em Subvolumes. Paths como `@snapshots/296/snapshot` entram em Snapshots.

Privilégio

Quando pede senha

Discovery/listagem usam ação Polkit sem senha para sessão ativa. Mount, umount, create/delete, readonly toggle, policy e rollback usam classes Polkit próprias e podem pedir autenticação administrativa.

Segurança

Paths seguros

Paths precisam ser absolutos e sem traversal. Cleanup só aceita roots gerenciados em `/tmp/btrfs-manager-*` ou `/run/user/<uid>/btrfs-manager`.

if path is "@snapshots" or ".snapshots":
    kind = SnapshotContainer
else if path contains "timeshift" or "snapper":
    kind = ExternalSnapshot(tool)
else if path contains "snapshots/" or ends with "/snapshot":
    kind = Snapshot
else:
    kind = Normal

Regra de review

Arquitetura antes de sintaxe

PRs devem explicar risco, testes, impacto em helper privilegiado, validação de paths, snapshots/rollback e screenshots quando tocar GUI.

Regra de regressão

Bug fix precisa de teste

Todo bug corrigido deve ganhar teste unitário, fixture ou cenário loopback que falharia antes da correção.

06 · Guia de Troubleshooting

Causa e efeito dos erros comuns

Polkit pede senha várias vezes ao abrir ou atualizar

Causa: serviço D-Bus/Polkit não instalado corretamente ou helper antigo ainda em uso. Correção atual: em builds instalados, a GUI usa apenas D-Bus; prompts devem aparecer apenas em mount/unmount/create/delete/toggle/policy write/rollback.

Discovery failed: Operation not permitted ao listar `/`

Causa: em alguns hosts, `btrfs subvolume list /` exige privilégio mesmo para leitura. Correção atual: a GUI instalada não executa listagem local; ela chama o serviço D-Bus, que autoriza discovery/listagem via Polkit.

missing field `kind` ao carregar inventário

Causa: GUI nova lendo JSON de helper antigo sem `SubvolumeKind`. Correção atual: desserialização usa `SubvolumeKind::Normal` como default para inventários antigos.

snapshot path is not accessible from current mountpoint

Causa: `/` pode estar montado como `subvol=@`, enquanto `btrfs subvolume list /` retorna caminhos relativos ao top-level. Correção: montar top-level read-only com `subvolid=5` e resolver o snapshot lá.

command failed: mount ["-o", "ro,subvolid=5", "..."]

Causa: source de `findmnt` pode vir como `/dev/mapper/cryptroot[/@]`; helper normaliza removendo sufixo `[/@]`. Se persistir, olhar stderr completo incluído em `HelperError::CommandFailed`.

Temporary mounts aparecem no lsblk

Causa: browse/top-level mounts ficam vivos enquanto a janela está aberta. Correção: fechar a GUI desmonta os mounts da sessão; botão eject ou `sudo target/debug/btrfs-manager-helper cleanup-managed-mounts` fazem cleanup global.

cargo check --features gui falha em gtk4.pc ou graphene-gobject-1.0.pc

Causa: faltam pacotes nativos de desenvolvimento. No Debian/Ubuntu: `libgtk-4-dev libadwaita-1-dev libgraphene-1.0-dev pkg-config`. No Arch: `gtk4 libadwaita graphene pkgconf`.
